# K8s Ops Agent — 项目介绍与优化建议

## 一、项目概述

**K8s Ops Agent** 是一个基于 Kubernetes 的智能运维 Agent 系统，核心理念是 **从传统脚本巡检进化到 AI 辅助运维**。项目包含两大部分：

1. **Todo 应用**（被运维对象）：一个基于 Flask + Redis 的待办事项 Web 应用，通过 Docker 容器化、Kubernetes 编排部署，作为运维 Agent 的"靶场"。
2. **运维 Agent**（运维工具）：一套 Shell 脚本工具集，覆盖日常巡检、日志分析、故障自愈、告警通知、AI 日志分析等运维场景。

```
传统运维：手动执行命令 → 人眼看结果 → 人处理
本项目：  Agent自动采集 → AI分析根因 → 自动修复 → 通知人
```

### 工作流

```
定时触发（crontab）
  → daily-inspection.sh 采集数据（节点/Pod/资源/事件）
  → 如果有告警：
      → alert.sh 发送飞书/钉钉/企微通知
      → auto-fix.sh 尝试自动修复（重启异常Pod/清理Evicted）
      → analyze.sh 调用LLM分析根因
  → report.sh 生成自然语言巡检报告
  → 存入 reports/ 目录
```

## 三、技术栈

| 层 | 技术 | 说明 |
|----|------|------|

## 四、模块详解

### 4.1 Todo 应用 (`app/`)

- **app.py**：Flask 应用，提供 5 个 RESTful 接口（GET/POST/PUT/DELETE /api/todos + /health）
- **templates/index.html**：单页面应用，包含添加、查看、完成、删除待办的完整交互，设计精美
- **requirements.txt**：依赖 flask 和 redis

### 4.2 运维工具层 (`ops/`)

| 脚本 | 功能 | 触发方式 |
|------|------|----------|
| daily-inspection.sh | 检查节点状态、异常Pod、高重启Pod、资源使用、Service健康、Warning事件 | crontab 定时 |
| log-analyzer.sh | 按日志级别统计、提取ERROR详情、异常关键词、HTTP状态码统计 | 手动/Agent调用 |
| auto-fix.sh | 重启CrashLoopBackOff Pod、清理Evicted Pod、检查PVC和节点状态 | crontab 定时 |
| alert.sh | 自动识别Webhook类型（飞书/钉钉/企微/通用），发送告警通知 | 其他脚本调用 |

### 4.3 AI 增强层 (`agent/`)

| 脚本 | 功能 | 依赖 |
|------|------|------|
| analyze.sh | 调用 OpenAI/Claude API 分析日志，返回根因、严重程度、修复建议 | LLM API Key |
| report.sh | 采集集群数据，生成带 emoji 的可读巡检报告 | kubectl |

### 4.4 配置层 (`config/`)

- **threshold.conf**：定义 CPU/内存/磁盘/重启次数/错误日志的告警阈值，以及 Webhook 地址和告警级别

### 4.5 K8s 部署 (`k8s/`)

- **app-deployment.yaml**：3 副本，ConfigMap/Secret 注入环境变量，健康检查探针
- **app-service.yaml**：NodePort 30080
- **redis-deployment.yaml**：单副本 Redis (alpine)，emptyDir 存储
- **redis-service.yaml**：ClusterIP
- **configmap.yaml** + **secret.yaml**：配置与密钥管理

## 五、项目亮点

1. **完整的运维闭环**：采集 → 分析 → 修复 → 通知，形成自动化运维链条
2. **AI 增强**：集成 LLM 进行日志根因分析，从传统脚本升级到智能运维
3. **多平台告警**：支持飞书、钉钉、企业微信、通用 Webhook 四种通知渠道
4. **K8s 最佳实践**：ConfigMap/Secret 注入、健康检查探针、多副本部署
5. **文档完善**：PRD、架构说明、故障排查手册一应俱全
6. **适合作为简历项目**：覆盖 Flask、Redis、Docker、Kubernetes、Nginx、Shell 等核心技术栈

## 六、优化建议

### 🔴 严重问题（建议优先修复）

#### 1. `threshold.conf` 与 `source` 不兼容
- **位置**：[ops/daily-inspection.sh](file:///c:/Users/cj/Desktop/k8s-ops-agent/ops/daily-inspection.sh#L16)、[ops/auto-fix.sh](file:///c:/Users/cj/Desktop/k8s-ops-agent/ops/auto-fix.sh#L13)、[ops/alert.sh](file:///c:/Users/cj/Desktop/k8s-ops-agent/ops/alert.sh#L12)
- **问题**：`threshold.conf` 是 INI 格式（含 `[section]`），但脚本用 `source` 当作 bash 脚本加载，`[threshold]` 等行会导致语法错误（虽然 `2>/dev/null` 掩盖了错误）
- **建议**：将配置文件改为纯 key=value 格式，或使用 `grep`/`awk` 解析 INI 格式

#### 2. `auto-fix.sh` 中 `FIXED` 计数器失效
- **位置**：[ops/auto-fix.sh](file:///c:/Users/cj/Desktop/k8s-ops-agent/ops/auto-fix.sh#L28-L29)
- **问题**：`while read` 在管道右侧运行于子 shell 中，`FIXED=$((FIXED + 1))` 的修改不会影响父 shell
- **建议**：使用进程替换 `while read ... done < <(echo "$CRASH_PODS")`，或使用临时文件计数

#### 3. Docker Compose 中 Nginx 构建路径错误
- **位置**：[docker/docker-compose.yml](file:///c:/Users/cj/Desktop/k8s-ops-agent/docker/docker-compose.yml#L21-L30)
- **问题**：`context: ../nginx` 但 `dockerfile: Dockerfile.nginx` 指向的是 `docker/Dockerfile.nginx` 而非 `nginx/` 目录下的文件，构建会失败
- **建议**：将 `Dockerfile.nginx` 移动到 `nginx/` 目录，或修改 compose 中 `dockerfile: ../docker/Dockerfile.nginx`

#### 4. Flask 生产环境 `debug=True`
- **位置**：[app/app.py](file:///c:/Users/cj/Desktop/k8s-ops-agent/app/app.py#L101)
- **问题**：`debug=True` 会启用 Werkzeug 调试器，允许远程代码执行，安全风险极高
- **建议**：从环境变量读取 debug 模式，生产环境设为 `False`；使用 Gunicorn/uWSGI 替代 Flask 开发服务器

#### 5. Redis 数据无持久化
- **位置**：[k8s/redis-deployment.yaml](file:///c:/Users/cj/Desktop/k8s-ops-agent/k8s/redis-deployment.yaml#L24-L25)
- **问题**：使用 `emptyDir: {}` 存储 Redis 数据，Pod 重启后数据全部丢失
- **建议**：改用 PVC（PersistentVolumeClaim）挂载持久存储，或至少配置 Redis AOF/RDB 持久化到 emptyDir 并在 Pod 删除前备份

### 🟡 中等问题

#### 6. Shell 脚本缺少错误处理
- **位置**：所有 `ops/*.sh` 和 `agent/*.sh`
- **问题**：没有 `set -euo pipefail`，kubectl 命令失败时脚本继续执行，可能产生误导性输出
- **建议**：在脚本开头添加 `set -euo pipefail`，并对关键命令添加错误检查

#### 7. `analyze.sh` 中 `jq` 依赖未检查
- **位置**：[agent/analyze.sh](file:///c:/Users/cj/Desktop/k8s-ops-agent/agent/analyze.sh#L75)
- **问题**：`jq -Rs` 用于 JSON 转义，但未检查 `jq` 是否安装；如果未安装，prompt 中的特殊字符会破坏 JSON 结构
- **建议**：在脚本开头检查 `jq` 可用性，或使用 Python 替代 JSON 构造

#### 8. `alert.sh` 中 JSON 消息未转义
- **位置**：[ops/alert.sh](file:///c:/Users/cj/Desktop/k8s-ops-agent/ops/alert.sh#L36-L39)
- **问题**：`MESSAGE` 包含换行符和特殊字符，直接拼接到 JSON 中可能导致 JSON 解析失败
- **建议**：使用 `jq` 构造 JSON 负载，或对消息内容进行转义

#### 9. `app.py` 中 `datetime.utcnow()` 已废弃
- **位置**：[app/app.py](file:///c:/Users/cj/Desktop/k8s-ops-agent/app/app.py#L64)
- **问题**：Python 3.12+ 中 `datetime.utcnow()` 已标记为废弃
- **建议**：改用 `datetime.now(datetime.UTC).isoformat()`

#### 10. 前端 XSS 防护仅在前端
- **位置**：[app/app.py](file:///c:/Users/cj/Desktop/k8s-ops-agent/app/app.py#L56-L57)
- **问题**：后端未对 `title` 做任何长度限制或内容校验，恶意用户可提交超长或含脚本的内容
- **建议**：后端添加 `title` 长度限制（如 200 字符），考虑使用 HTML 转义库

#### 11. 健康检查裸 `except`
- **位置**：[app/app.py](file:///c:/Users/cj/Desktop/k8s-ops-agent/app/app.py#L32)
- **问题**：`except:` 会捕获所有异常包括 `KeyboardInterrupt` 和 `SystemExit`
- **建议**：改为 `except redis.RedisError:` 或至少 `except Exception:`

#### 12. K8s 资源缺少 limits/requests
- **位置**：所有 `k8s/*-deployment.yaml`
- **问题**：没有设置 CPU/内存的 `resources.requests` 和 `resources.limits`，可能导致资源争抢或 OOM
- **建议**：为所有容器添加合理的资源限制和请求值

#### 13. `imagePullPolicy: Never` 限制集群适用性
- **位置**：[k8s/app-deployment.yaml](file:///c:/Users/cj/Desktop/k8s-ops-agent/k8s/app-deployment.yaml#L18)
- **问题**：`Never` 策略仅适用于 minikube 等本地环境，无法在标准 K8s 集群中使用
- **建议**：改为 `IfNotPresent`，并将镜像推送到镜像仓库

### 🟢 改进建议

#### 14. 缺少测试
- **问题**：整个项目没有任何自动化测试（单元测试、集成测试、Shell 脚本测试）
- **建议**：为 Flask 应用添加 pytest 测试；为 Shell 脚本添加 bats 或 shellspec 测试

#### 15. 缺少 CI/CD
- **问题**：没有 GitHub Actions / GitLab CI 等 CI/CD 流水线
- **建议**：添加 CI 流水线实现自动构建、测试、镜像推送

#### 16. 缺少 `.dockerignore`
- **问题**：构建镜像时会把 `reports/`、`__pycache__/`、`.git/` 等不必要文件也打包进去
- **建议**：添加 `.dockerignore` 排除不必要文件，减小镜像体积

#### 17. 容器未使用非 root 用户
- **位置**：[docker/Dockerfile](file:///c:/Users/cj/Desktop/k8s-ops-agent/docker/Dockerfile)
- **问题**：容器默认以 root 运行，存在安全风险
- **建议**：添加 `RUN useradd` 和 `USER` 指令，以非 root 用户运行

#### 18. Nginx 配置冗余
- **位置**：[nginx/nginx.conf](file:///c:/Users/cj/Desktop/k8s-ops-agent/nginx/nginx.conf#L18-L26)
- **问题**：`location /` 和 `location /api/` 和 `location /health` 都代理到同一个 upstream，可以合并
- **建议**：除非有特殊需求，否则一个 `location /` 即可覆盖所有路径

#### 19. 缺少日志轮转
- **问题**：`reports/` 目录下的巡检报告和自愈日志会无限增长
- **建议**：添加日志清理策略（如保留最近 30 天），或使用 logrotate

#### 20. 缺少 HPA（水平自动扩缩容）
- **问题**：Flask 应用固定 3 副本，无法根据负载自动伸缩
- **建议**：添加 HorizontalPodAutoscaler 根据 CPU/内存使用率自动调整副本数

#### 21. 缺少 NetworkPolicy
- **问题**：没有网络策略限制 Pod 间通信
- **建议**：添加 NetworkPolicy 限制只有 Flask 可以访问 Redis，提高安全性

#### 22. 缺少 PodDisruptionBudget
- **问题**：自愿性中断（如节点维护）可能导致所有副本同时不可用
- **建议**：添加 PDB 确保至少 1 个副本始终可用

#### 23. 巡检脚本缺少 CPU/内存阈值告警联动
- **位置**：[ops/daily-inspection.sh](file:///c:/Users/cj/Desktop/k8s-ops-agent/ops/daily-inspection.sh)
- **问题**：`threshold.conf` 中定义了 `cpu_warn`、`mem_warn` 等阈值，但巡检脚本未实际使用这些阈值进行比较
- **建议**：在资源使用检查部分解析 `kubectl top` 输出并与阈值比较，超出时触发告警

#### 24. Docker 镜像未使用多阶段构建
- **位置**：[docker/Dockerfile](file:///c:/Users/cj/Desktop/k8s-ops-agent/docker/Dockerfile)
- **问题**：最终镜像包含 pip 缓存等构建依赖
- **建议**：使用多阶段构建减小镜像体积，或添加 `--no-cache-dir` 和清理步骤

#### 25. 缺少 API 限流
- **位置**：[app/app.py](file:///c:/Users/cj/Desktop/k8s-ops-agent/app/app.py)
- **问题**：没有请求频率限制，可能被恶意调用打垮
- **建议**：使用 Flask-Limiter 添加速率限制
